# AWS Infrastructure — Terraform

Production-grade, multi-module AWS infrastructure built with Terraform. Designed for high availability, security, and scalability across multiple environments.

---

## Architecture

```
                          INTERNET
                              │
               ┌──────────────┴──────────────┐
               │                             │
          port 80/443                     port 22
          (user traffic)              (engineer SSH)
               │                             │
    ┌─────────────────┐             ┌────────────────┐
    │       ALB       │             │    Bastion     │
    │  (public subnet)│             │ (public subnet)│
    └────────┬────────┘             └───────┬────────┘
             │                             │
    ┌────────┴────────────────────────────-┘
    │         Private Subnet (us-east-2a / us-east-2b)
    │
    ├── App Server 1 (us-east-2a)  ──┐
    ├── App Server 2 (us-east-2b)  ──┤──→ RDS MySQL (private subnet)
    └── App Server 3 (us-east-2a)  ──┘
              │
              └──→ S3 Bucket (via IAM Role — no hardcoded keys)
```

---

## Modules

### VPC
Networking foundation. Every other module depends on this. Two public and two private subnets spread across `us-east-2a` and `us-east-2b` for high availability. Internet Gateway for public subnets, NAT Gateway for private subnet outbound traffic, with separate route tables for each subnet type.

### EC2
Bastion host in public subnet as the sole SSH entry point for engineers. Three app servers in private subnets, distributed across AZs using modulo-based subnet selection. SSH key pair generated by Terraform itself using `tls_private_key` — no manual key management.

### RDS
MySQL 8.0 in private subnets with a DB subnet group spanning both AZs for failover readiness. Port 3306 open only to the app server security group — not to any IP range. `publicly_accessible = false` enforced.

### ALB
Internet-facing Application Load Balancer spanning both public subnets. Target group registers all three app servers with health checks on `/health` every 30 seconds. Unhealthy instances are automatically removed from rotation.

### S3
Private bucket with public access fully blocked (all four `block_public_access` settings). AES256 server-side encryption at rest. Versioning enabled for file recovery.

### IAM
EC2 IAM role with a least-privilege S3 policy (`GetObject`, `PutObject`, `DeleteObject`, `ListBucket`). Instance profile attached to all EC2 instances — credentials rotate automatically, zero hardcoded access keys anywhere in the codebase.

## Environments

```
environments/
├── dev/
│   ├── main.tf           # calls all modules
│   ├── variables.tf      # variable declarations
│   └── terraform.tfvars  # actual values — only file that changes per env
```

The module code never changes between environments. Only `terraform.tfvars` changes. Scaling from 3 to 10 app servers is a single line change: `app_server_count = 10`.

## Security design

| Layer | Implementation |
|---|---|
| Network | App servers and RDS in private subnets — no public IPs |
| Access control | Security groups use SG references, not IP ranges |
| SSH | Bastion-only entry; app-sg allows SSH from bastion-sg, not 0.0.0.0/0 |
| Database | Port 3306 open only to app-sg |
| Credentials | IAM instance profile on EC2 — no access keys in code or environment |
| Storage | S3 fully private, encrypted, versioned |

## What I learned debugging this


### NAT Gateway placement
My first design placed the NAT Gateway in a private subnet. Traffic from private instances couldn't reach the internet because NAT itself needs an IGW route to forward traffic. Moved NAT to a public subnet where the IGW route exists.

### Security group circular dependency
Tried to allow ALB traffic to app servers by referencing `alb_sg_id` inside the EC2 module, and `app_sg_id` inside the ALB module — creating a circular dependency. Resolved by keeping the existing VPC CIDR rule (`10.0.0.0/16`) on the app server, since ALB already lives inside the VPC.

### IAM role vs instance profile
Attempted to attach an IAM role directly to an EC2 instance. AWS does not allow this — EC2 requires an instance profile as a wrapper around the role. Added `aws_iam_instance_profile` as the intermediary.

## Stack

- **IaC** — Terraform 1.6+, HCL
- **Cloud** — AWS (us-east-2)
- **Compute** — EC2 (t3.micro), Auto-distributed across AZs
- **Database** — RDS MySQL 8.0 (db.t3.micro)
- **Networking** — VPC, IGW, NAT Gateway, ALB
- **Storage** — S3 (encrypted, versioned)
- **Security** — IAM roles, Security Groups, Private subnets

## What's next

**Remote state** — moving `terraform.tfstate` from local disk to S3 with DynamoDB state locking. This allows teams to collaborate on the same infrastructure without state conflicts.

**Puppet** — configuration management on top of this infrastructure. Terraform provisions the servers; Puppet configures them — installing packages, managing services, enforcing configuration drift.

## Usage

```bash
cd environments/dev
terraform init
terraform plan
terraform apply
```

```bash
# SSH into infrastructure
ssh -i modules/ec2/dev-key.pem ec2-user@<bastion_public_ip>

# from bastion into app server
ssh -i dev-key.pem ec2-user@<app_private_ip>
```
